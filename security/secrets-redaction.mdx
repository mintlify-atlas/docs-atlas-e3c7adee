title: "Redact secrets from build logs"
description: "Automatically redact sensitive values from Buildkite job logs using the redacted-vars agent flag or the buildkite-agent redactor add command mid-job."
---

Buildkite agents can automatically detect and redact sensitive values from job log output before the logs are uploaded to Buildkite. Redaction operates as a streaming replacement pass on all log bytes written by a job: any matching value is replaced with the literal string `[REDACTED]`. This protects secrets that are exported to the environment from appearing in build logs even when commands inadvertently print them.

## The `--redacted-vars` flag

When the agent starts, you configure which environment variable names are treated as containing secrets. Any variable whose name matches one of the configured glob patterns has its value added to the redactor before the job runs.

<ParamField path="--redacted-vars" type="string[]">
  A list of glob patterns matched against environment variable names. Variables whose names match a pattern and whose values are at least 6 characters long have their values redacted from log output. Set via `BUILDKITE_REDACTED_VARS`.
</ParamField>

The default patterns are:

```
*_PASSWORD
*_SECRET
*_TOKEN
*_PRIVATE_KEY
*_ACCESS_KEY
*_SECRET_KEY
*_CONNECTION_STRING
*_API_KEY
```

These defaults are additive — specifying additional patterns in your agent configuration does not remove the defaults unless you explicitly override the full list.

<Note>
  Values shorter than 6 characters are never redacted, even if the variable name matches. Short values are too common in non-sensitive output (e.g. `none`, `true`, `yes`) and would produce excessive false-positive redactions. A warning is logged for matched variables with short values.
</Note>

### Customising patterns

Override or extend the list in your agent config file or via the environment variable:

```cfg
# buildkite-agent.cfg
redacted-vars="*_PASSWORD,*_SECRET,*_TOKEN,*_PRIVATE_KEY,*_ACCESS_KEY,*_SECRET_KEY,*_CONNECTION_STRING,*_API_KEY,MY_CUSTOM_*"
```

Or on the command line:

```bash
buildkite-agent start \
  --redacted-vars "*_PASSWORD" \
  --redacted-vars "*_SECRET" \
  --redacted-vars "*_TOKEN" \
  --redacted-vars "MY_CUSTOM_*"
```

## Adding values to the redactor mid-job

If a job fetches secrets dynamically (for example, from a secrets manager), those values are not in the environment when the job starts and therefore are not automatically redacted. Use `buildkite-agent redactor add` to register new values with the log redactor while the job is running.

```bash
# Redact a raw file (the entire trimmed file content is registered)
buildkite-agent redactor add ./my-private-key

# Redact a value read from stdin
echo "my-dynamic-secret" | buildkite-agent redactor add

# Redact multiple secrets from a flat JSON object
echo '{"db_password":"secret1","api_token":"secret2"}' \
  | buildkite-agent redactor add --format json

# Redact from a JSON file
buildkite-agent redactor add --format json ./secrets.json
```

<ParamField path="--format" type="string" default="none">
  Controls how the input is parsed. `none` treats the entire trimmed input as a single string to redact. `json` parses the input as a flat JSON object and registers each value individually. Set via `BUILDKITE_AGENT_REDACT_ADD_FORMAT`.
</ParamField>

<Warning>
  `buildkite-agent redactor add` only affects log output produced **after** the command runs. Call it as early as possible in your job, before any commands that might print the secret.
</Warning>

### In-job example

```bash
# Fetch a secret from AWS Secrets Manager
MY_SECRET=$(aws secretsmanager get-secret-value \
  --secret-id prod/my-app/api-key \
  --query SecretString \
  --output text)

# Immediately register it with the redactor
echo "$MY_SECRET" | buildkite-agent redactor add

# Now safe to use - the value will appear as [REDACTED] in logs
curl -H "Authorization: Bearer $MY_SECRET" https://api.example.com
```

## Buildkite Pipelines Secrets integration

If you use Buildkite's built-in Pipelines Secrets, use `buildkite-agent secret get` to retrieve secrets during a job. Secret values fetched this way are **automatically registered with the redactor** — you do not need to call `buildkite-agent redactor add` separately.

```bash
# Fetch a single secret (printed to stdout, automatically redacted from logs)
MY_KEY=$(buildkite-agent secret get deploy_key)

# Fetch multiple secrets formatted as shell variable assignments
eval "$(buildkite-agent secret get --format env deploy_key github_token)"
echo "$DEPLOY_KEY"        # value appears as [REDACTED] in logs
echo "$GITHUB_TOKEN"      # value appears as [REDACTED] in logs

# Fetch as JSON
buildkite-agent secret get --format json deploy_key github_token
```

<ParamField path="--format" type="string" default="default">
  Output format. `default` prints the raw value for a single key, or JSON for multiple keys. `json` always produces a JSON object. `env` produces `KEY="value"` shell assignments suitable for `eval`. Set via `BUILDKITE_AGENT_SECRET_GET_FORMAT`.
</ParamField>

<ParamField path="--skip-redaction" type="boolean" default="false">
  Bypass automatic redaction of the fetched secret. The secret value will appear in logs if any subsequent command prints it. Use only for non-sensitive values or during debugging. Set via `BUILDKITE_AGENT_SECRET_GET_SKIP_SECRET_REDACTION`.
</ParamField>

<Note>
  Secret keys in `buildkite-agent secret get` are case-insensitive. `deploy_key` and `DEPLOY_KEY` retrieve the same secret.
</Note>

## How redaction works

The redactor is a streaming byte-level replacer wrapping the job's log output writer. As bytes flow from the job process, the replacer scans for occurrences of any registered needle string and substitutes `[REDACTED]` in their place. Because it operates on the raw byte stream, it catches secrets regardless of how they appear (direct echo, command substitution, tool output, etc.).

The minimum needle length of 6 characters (`LengthMin = 6` in the source) prevents pathologically short strings from causing excessive false-positive redactions in log output.

## Best practices

- **Use `secret get` for Buildkite-managed secrets.** It handles redaction automatically and keeps your pipeline YAML free of secret names.
- **Call `redactor add` immediately after fetching any external secret.** Do not wait until later in the script — any output between the fetch and the `redactor add` call could expose the value.
- **Avoid `--skip-redaction` in production pipelines.** It is intended only for debugging.
- **Audit your `--redacted-vars` patterns regularly.** When you add new secret-bearing environment variables with names that do not match the defaults, add their patterns to your agent configuration.
- **Do not rely on redaction as the only control.** Redaction is a last line of defence. Use secret scoping, least-privilege IAM policies, and short-lived credentials to limit secret exposure in the first place.
